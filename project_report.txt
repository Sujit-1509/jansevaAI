1. EXECUTIVE SUMMARY & SYSTEM LIFECYCLE
--------------------------------------------------------------------------------
JanSevaAI is a fully secure, serverless Civic Engagement platform designed to bridge the gap between citizens and municipal authorities. By leveraging advanced Computer Vision (YOLOv8) and Generative LLMs (Amazon Bedrock), the platform completely automates the classification, assessment, description, and routing of civic complaints (e.g., potholes, garbage accumulation, broken streetlights, waterlogging) strictly through a single photo upload.

The lifecycle involves:
1. Citizen authentication via Passwordless SMS OTP (Secure JWT).
2. Frontend requesting a highly-secure S3 Presigned URL to bypass standard API Gateway payload limits.
3. The image upload triggering an asynchronous processing Lambda connected to an EC2 GPU server and Bedrock APIs.
4. Intelligent routing through DynamodDB logic, sending emails to administrators, and tracking SLA deadlines based on calculated issue severity.


--------------------------------------------------------------------------------
2. FRONTEND ARCHITECTURE DEEP-DIVE
--------------------------------------------------------------------------------
Built as a highly optimized React 19 Single Page Application (SPA), routed by React Router v7. 
- Deployment: Fully static hosting on an Amazon S3 Bucket distributed globally via Amazon CloudFront (CDN) to ensure <50ms TTFB (Time To First Byte) anywhere in the country.
- Styling: Zero heavy CSS frameworks (No Tailwind/Bootstrap). The entire design system is built on vanilla CSS modules using CSS Variables for precise Glassmorphism, theming, and flexbox grid systems, ensuring extremely low bundle sizes.
- UX Features: 
  - GPS Metadata Extraction: Prompts exactly when required to pull HTML5 Geolocation (Lat/Lng) coords.
  - Multi-Stage State Preservation: Uses SessionStorage to keep intermediate complaint drafts alive even if the user refreshes or closes the tab accidentally.
  - Granular Access Control: Custom UI dashboards routing dynamically based on the decoded JWT role ('citizen', 'worker', 'admin').
  - Error Boundaries: Top-level React error boundaries preventing white-screens on latent API failures.


--------------------------------------------------------------------------------
3. EXHAUSTIVE MICROSERVICES CATALOG (13 AWS LAMBDA FUNCTIONS)
--------------------------------------------------------------------------------
Every piece of backend compute is handled by independent serverless Python 3.12 AWS Lambda functions, masked securely behind Amazon API Gateway. None of the functions explicitly rely on heavy external libraries like `pyjwt` or `SQLAlchemy`, ensuring blazing-fast <50ms cold-start times. 

Below is the absolute granular breakdown of every Lambda and its usage.

[1] Authentication Service (`backend/auth`)
- API Route: POST `/auth/send-otp`, POST `/auth/verify-otp`, GET `/auth/verify-token`
- Usage: Controls the entire passwordless login flow. 
- Send-OTP: Connects to Amazon SNS (Simple Notification Service) and Amazon Pinpoint. Generates a random 6-digit integer and saves the tuple (phone, OTP, TTL) into the `Users` DynamoDB table. It sends SMS text directly to the user's phone.
- Verify-OTP: Checks the user-provided code against the DynamoDB record. Crucially, if the requested role is 'worker', the code makes a cross-check query against the `Workers` DynamoDB table to ensure the phone number is actually pre-registered by an admin. 
- JWT Construction: Manually assembles a JSON Web Token using standard library `hashlib` and `hmac` (SHA-256). The secret is injected securely via Lambda Environment Variables (`JWT_SECRET`). Tokens are stamped with a 7-day expiration (`exp`) and bypass the need for external session databases entirely.

[2] Secure Upload URL Generator (`backend/generate_upload_url`)
- API Route: POST `/upload/presign`
- Usage: Due to API Gateway's 10MB payload limit, binary images cannot be sent via traditional POST requests reliably. 
- Execution: Accepts a JSON body mapping `fileName` and `fileType`. It generates a unique `uuid4` which becomes the `incident_id`. Through `boto3`, it queries the specific `JanSevaAI-images` S3 bucket, commanding S3 to generate a "Presigned PUT URL". This URL is cryptographically signed and expires strictly in 5 minutes (300 seconds), heavily restricting unauthorized payload injections. The frontend natively PUTs the binary image directly to this S3 Link.

[3] Core AI Processing Engine (`backend/process_image`)
- API Route: NONE. (Triggered internally via S3 ObjectCreated Events)
- Usage: This is the heavyweight background worker function that orchestrates the AI pipelines.
- Step A: Parses the S3 bucket and object key from the event payload.
- Step B (YOLO Classification): Executes an HTTP POST request via `requests` to an external fixed IP EC2 Instance running a FastAPI YOLOv8 model trained expressly to classify (pothole, garbage, streetlight, water).
- Step C (The Vision Fallback Routine): If the YOLO model's confidence is 0.0 or the object class is 'Unknown', the Lambda invokes the cutting-edge Amazon Nova Lite (`amazon.nova-lite-v1:0`) Multimodal LLM natively via `boto3 bedrock-runtime`. It feeds the binary image bytes directly to Nova with a stringent zero-shot prompt forcing it into one of the 4 valid categories.
- Step D: Calculates numerical 'Severity' mapping arrays based on probability floats.
- Step E: Triggers an Amazon Bedrock generative invocation (using Anthropic Claude-v3 Haiku) injecting context parameters to artificially generate a beautiful, syntactically perfect, formal "Description of Complaint" paragraph, relieving the user of typing duty.
- Step F: Computes the fundamental Base Priority Score referencing location, timestamp, and severity tiers.
- Step G: Structures a complex JSON dictionary and natively executes a `put_item` onto the primary DynamoDB `Complaints` table.
- Step H: Instantiates an Amazon SES (Simple Email Service) client and blasts an emergency HTML/Text email to predefined administrator distribution lists detailing the exact nature of the freshly captured incident.

[4] Finalize Complaint Submission (`backend/submit_complaint`)
- API Route: POST `/complaints`
- Usage: Called by the frontend just after the presigned upload succeeds. 
- Execution: Because `process_image` runs concurrently asynchronously, this Lambda ensures the user's manual inputs (Address context, Notes, precise GPS Latitude/Longitude metadata) are successfully merged into the DynamoDB record.
- Security: Forces the `user_phone` parameter explicitly from the decoded JWT. A malicious actor cannot modify the HTTP payload to submit a complaint under someone else's phone number.

[5] Get User Complaints (`backend/get_user_complaints`)
- API Route: GET `/complaints`
- Usage: Fetches listed complaints dynamically.
- Execution: If the requester's JWT role is 'citizen', the Lambda strictly enforces query arguments replacing the `phone` parameter with their own matching JWT phone. It completely prevents lateral data leakage (IDOR). 
- Database Logic: It uses a Global Secondary Index (GSI) named `user_phone-index` to perform an extraordinarily fast O(log N) `query()` rather than a highly expensive O(N) `scan()`. It dynamically constructs `boto3.dynamodb.conditions.Attr` filters if the user is filtering by `status` or combined `category` buckets (e.g. mapping frontend 'lighting' to backend 'streetlight').
- Fallback: Admins query this endpoint with no phone filter, triggering an authorized full-table `scan()`.

[6] Get Singular Complaint Detail (`backend/get_complaint`)
- API Route: GET `/complaints/{id}`
- Usage: Powers the detailed single-page React view. Uses simple `Key={'incident_id': id}` `get_item` lookups.

[7] Proximity Mapping Algorithm (`backend/get_nearby_complaints`)
- API Route: GET `/complaints/nearby?lat=X&lng=Y&radius=M`
- Usage: Powers the map/feed to show citizens complaints exclusively within their geolocation radius. Features an embedded Python implementation of the Haversine Formula.
- Algorithm: Queries DynamoDB items. Extracts lat/lng strings. Parses math: `R = 6371000m (Radius of Earth)`. Calculates exact scalar meter distances `math.sin(dlat/2)**2 + math.cos(lat1) * math.cos(lat2) * ...`. Drops points completely if distance exceeds `radius_m`. Excludes 'resolved' claims to shrink output bandwidth natively. Sorts cleanly and truncates `array[:20]` before JSON serialization.

[8] Role-Gated Status Updates & Proofs (`backend/update_complaint_status`)
- API Route: PATCH `/complaints/{id}/status`
- Usage: Handles state synchronization across users, workers, and admins. 
- Execution: Relies heavily on Dynamic DynamoDB UpdateExpressions mapping `#st = :status, list_append(if_not_exists(status_history, :empty), :entry)`. 
- Logic Gates: Workers are programmatically blocked from arbitrarily changing statuses globally; they are limited to state-machine transitions: ('accepted'/'in_progress', 'rejected', or 'resolved').
- Audit Chains: Every status change natively injects a JSON blob into the `status_history` LIST object detailing the Actor Phone, Timestamp, and any `resolve_location` metadata or `proof_s3_key` (pictures of repaired potholes) establishing proof-of-work.

[9] Mass Administrative Bulk Operations (`backend/bulk_update`)
- API Route: POST `/complaints/bulk`
- Usage: Admin operational dashboard utility. Allows mass 'resolve', 'close', or mass worker 'assign'.
- Execution: Accepts JSON Arrays up to `100` incident_ids. Loops through `dynamodb.update_item()`. Critically, catching `ClientError` inside the loop ensures that if 1 write fails, the remaining 99 assignments succeed, returning a parsed `{updated: [], failed: []}` dict back to the frontend for precise UI error toast notifications.

[10] Community Upvoting Engine (`backend/upvote_complaint`)
- API Route: POST `/complaints/{id}/upvote`
- Usage: Community engagement metric endpoint. 
- Algorithmic Integration: Utilizing DynamoDB mathematical operators (`SET upvotes = upvotes + :inc`), it natively increments the base upvote count. Simultaneously, it manipulates the backend `priorityScore` logic (`+ :inc_prio` = 2 points). This instantly pushes highly-demanded civic requests structurally higher in worker queues without rewriting the entire record.

[11] Worker Assignment & SLA Computations (`backend/assign_complaint`)
- API Route: POST `/complaints/{id}/assign`
- Usage: Specific worker task allocations (Admin dashboard).
- Execution: Reads the DynamoDB record. Computes critical Service Level Agreement (SLA) deadlines dynamically based on categorical urgency metrics:
  -> Waterlogging: Extremely Urgent (24 Hours)
  -> Potholes: High Priority (48 Hours)
  -> Garbage: Moderate (72 Hours)
  -> Streetlights: Standard (96 Hours)
- Computes `datetime.timedelta(hours=X)` and registers this UTC ISO8601 string permanently via `sla_deadline`.

[12] Administrative Purge (`backend/delete_complaint`)
- API Route: DELETE `/complaints/{id}`
- Usage: Destroys falsified data. Admin deletes implicitly authorized. Citizen deletes checked rigidly against the `user_phone` equivalent in the DynamoDB `Item`.

[13] Workforce Human Resources Management (`backend/manage_workers`)
- API Route: ANY `/workers/*` (Usually GET/POST/DELETE)
- Usage: Orchestrates the dedicated `Workers` DynamoDB table. Contains custom `POST handle_add_worker` maintaining Name, Phone, and Department constraints. This acts as the single source of truth authorizing field agents to bypass the OTP flow and access the Field Worker Dashboard views.

4. DATABASE SCHEMA DESIGN (AMAZON DYNAMODB)
--------------------------------------------------------------------------------
The database entirely leverages DynamoDB's schema-less document structural modeling to maximize NoSQL flexibility while enforcing structural indexes via AWS CDK/Console definitions.

* Table: Complaints 
  Primary Key (HASH): `incident_id` (String | UUIDv4)
  GSI 1 (HASH)      : `user_phone` (String) -> Projection: ALL
  - Dynamic Documents: The system injects arrays (`status_history`, `images`) inside standard rows. Time Series logic is implemented entirely by mapping string `timestamp` and `resolved_at` headers to ISO8601 formatting for absolute cross-platform datetime lexicographical sorting.

* Table: Users
  Primary Key (HASH): `phone` (String)
  - Purpose: Tracks the ephemeral state of Authentication. 
  - Sub-features: Enforces Amazon DynamoDB TTL (Time-To-Live). The `expires_at` column is read continuously by native AWS daemon processes wiping out expired OTP pins exactly after 5 minutes, saving extensive manual cron-script overheads.

* Table: Workers
  Primary Key (HASH): `phone` (String)
  - Purpose: Maps phone numbers to 'Role: Worker' allowing OTP bypass checks internally on JWT generation.

5. SECURITY POSTURE & AWS INTEGRATION 
--------------------------------------------------------------------------------
1. CORS Protocol: All 13 Lambdas possess an explicit `OPTIONS` preflight condition returning exact `Access-Control-Allow-Origin: *` headers, rendering browser-based CORS errors obsolete.
2. S3 Permissions: The target bucket uses `Block Public Access: OFF` combined precisely with `s3:PutObject` strictly restricted to presigned URL grants mapping, rendering bucket injections impossible without prior Lambda authorization.
3. IAM Scoping: Each AWS Lambda assumes granular Execution roles. The `process_image` lambda singularly possesses `ses:SendEmail` constraints, meaning a hijacked frontend lambda cannot arbitrarily spam emails.
