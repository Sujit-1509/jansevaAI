1. EXECUTIVE SUMMARY & SYSTEM LIFECYCLE
--------------------------------------------------------------------------------
JanSevaAI is a fully secure, serverless Civic Engagement platform designed to bridge the gap between citizens and municipal authorities. By leveraging advanced Computer Vision (YOLOv8) and Generative LLMs (Amazon Bedrock), the platform completely automates the classification, assessment, description, and routing of civic complaints (e.g., potholes, garbage accumulation, broken streetlights, waterlogging) strictly through a single photo upload.

The lifecycle involves:
1. Citizen authentication via Passwordless SMS OTP (Secure JWT).
2. Frontend requesting a highly-secure S3 Presigned URL to bypass standard API Gateway payload limits.
3. The image upload triggering an asynchronous processing Lambda connected to an EC2 GPU server and Bedrock APIs.
4. Intelligent routing through DynamodDB logic, sending emails to administrators, and tracking SLA deadlines based on calculated issue severity.


--------------------------------------------------------------------------------
2. SOCIO-ECONOMIC IMPACT & PROBLEM STATEMENT
--------------------------------------------------------------------------------
Current municipal complaint systems in India suffer from "The Void of Silence"—where citizens report issues via phone or paper and never receive updates. 

JanSevaAI addresses three critical pain points:
- **Triage Latency**: Human operators take 24-48 hours just to categorize a complaint. JanSevaAI does it in <5 seconds.
- **Reporting Friction**: Users often don't know the exact category or technical department for a pothole or a leaking pipe. Our "Photo-to-Submission" pipeline removes all cognitive load.
- **Accountability Gap**: Static PDF reports are replaced by a live, dynamic dashboard where every status change is timestamped and attributed to a specific worker or admin.

3. FRONTEND ARCHITECTURE & UX PHILOSOPHY
--------------------------------------------------------------------------------
Built as a highly optimized React 19 Single Page Application (SPA), routed by React Router v7. 
- **Premium Design System**: The UI follows a "Glassmorphism" aesthetic, utilizing subtle backdrops, rich gradients, and micro-animations to provide a premium feel that rivals high-end private sector applications.
- **Dynamic Dashboarding**: Integrated Recharts library provides real-time visualization of municipal health, categorical hotspots, and department-wise SLA adherence rates.
- **Deployment**: Fully static hosting on an Amazon S3 Bucket distributed globally via Amazon CloudFront (CDN) to ensure <50ms TTFB anywhere in the country.
- **Styling**: Zero heavy CSS frameworks (No Tailwind/Bootstrap). The entire design system is built on vanilla CSS modules using CSS Variables for precise theming and flexbox grid systems.
- **UX Features**: 
  - GPS Metadata Extraction: Prompts exactly when required to pull HTML5 Geolocation (Lat/Lng) coords.
  - Multi-Stage State Preservation: Uses SessionStorage to keep intermediate complaint drafts alive even if the user refreshes or closes the tab accidentally.
  - Granular Access Control: Custom UI dashboards routing dynamically based on the decoded JWT role ('citizen', 'worker', 'admin').


--------------------------------------------------------------------------------
4. EXHAUSTIVE MICROSERVICES CATALOG (13 AWS LAMBDA FUNCTIONS)
--------------------------------------------------------------------------------
Every piece of backend compute is handled by independent serverless Python 3.12 AWS Lambda functions, masked securely behind Amazon API Gateway. Every function uses standard library `hashlib` and `hmac` for JWT verification, ensuring blazing-fast <50ms cold-start times.

[1] Authentication Service (`backend/auth`)
- API Route: POST `/auth/send-otp`, POST `/auth/verify-otp`, GET `/auth/verify-token`
- Usage: Controls the entire passwordless login flow. 
- Send-OTP: Connects to Amazon SNS (Simple Notification Service) to send OTP via SMS. Saves (phone, OTP, TTL) into the `Users` DynamoDB table.
- Verify-OTP: Checks the user-provided code against the DynamoDB record. For workers, it cross-checks the `Workers` table.
- JWT Construction: Manually assembles a JSON Web Token (SHA-256). Secret is injected via environment variables.

[2] Secure Upload URL Generator (`backend/generate_upload_url`)
- API Route: POST `/upload/presign`
- Usage: Bypass API Gateway's 10MB payload limit by generating a unique `uuid4` (incident_id) and a 5-minute S3 Presigned PUT URL.

[3] Core AI Processing Engine (`backend/process_image`)
- API Route: NONE (Triggered by S3 Events)
- YOLO Classification: Executes an HTTP POST to an EC2 GPU Instance running a FastAPI YOLOv8 model (categories: pothole, garbage, streetlight, water).
- Vision Fallback: If YOLO confidence is low, invokes Amazon Nova Lite (`amazon.nova-lite-v1:0`) via Bedrock to classify the image.
- Bedrock Generative Description: Uses Claude-v3 Haiku to generate a professional, formal description of the incident from visual context.
- Priority Computation: Computes Base Priority Score based on category, severity, and upvote density.
- Notifications: Blasts an emergency email via Amazon SES to administrators once a "High Priority" incident is recorded.

[4] Finalize Complaint Submission (`backend/submit_complaint`)
- Merges user's manual inputs (Address, Notes, GPS) into the AI-generated record.

[5] Get User Complaints (`backend/get_user_complaints`)
- Uses GSI `user_phone-index` for O(log N) queries. Strictly enforces phone-mapping to prevent IDOR attacks.

[6] Get Singular Complaint Detail (`backend/get_complaint`)
- Powers the detailed single-page React view with timeline history.

[7] Proximity Mapping Algorithm (`backend/get_nearby_complaints`)
- Uses Haversine Formula to calculate distance in meters between user and complaint. Sorts by proximity.

[8] Role-Gated Status Updates & Proofs (`backend/update_complaint_status`)
- Forces state-machine transitions: ('assigned' -> 'in_progress' -> 'resolved'). Requires image proof for resolution.

[9] Mass Administrative Bulk Operations (`backend/bulk_update`)
- Admin operational dashboard utility for mass 'resolve' or mass worker 'assign'.

[10] Community Upvoting Engine (`backend/upvote_complaint`)
- Community engagement metric. Each upvote increments the priority score algorithmically.

[11] Worker Assignment & SLA Computations (`backend/assign_complaint`)
- Custom SLA Logic: Water (24h), Potholes (48h), Garbage (72h), Lighting (96h).

[12] Administrative Purge (`backend/delete_complaint`)
- Final gatekeeper for data deletion, gated by admin role or owner phone matching.

[13] Workforce Human Resources Management (`backend/manage_workers`)
- Orchestrates the `Workers` DynamoDB table, allowing specific departmental routing.


--------------------------------------------------------------------------------
5. TECHNICAL CHALLENGES & SOLUTIONS
--------------------------------------------------------------------------------
- **API Payload Limits**: API Gateway restricts file uploads to 10MB. We solved this by using S3 Presigned URLs, moving binary data directly from the browser to S3, reducing backend compute costs and latency.
- **AI Latency**: YOLO inference can take time. We implemented an asynchronous polling mechanism in the React frontend, using a status-state in DynamoDB to inform the user of progress (detecting -> severity -> finalizing).
- **Concurrency**: Multiple workers updating the same status could lead to race conditions. We utilized DynamoDB's `UpdateExpression` with conditional checks to ensure status history is appended atomically.


--------------------------------------------------------------------------------
6. DATABASE SCHEMA DESIGN (AMAZON DYNAMODB)
--------------------------------------------------------------------------------
* Table: Complaints 
  - Primary Key: `incident_id` (HASH)
  - GSI: `user_phone` (HASH) for citizen history.
  - Structure: Flexible document model stores `status_history` as an array of JSON objects.

* Table: Users & Workers
  - Uses TTL (Time-To-Live) on `expires_at` to automatically purge single-use OTPs, reducing administrative overhead.
